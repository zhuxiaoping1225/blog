# 1.开篇词：设计初衷及理论科普

# 一、 前言

现如今，企业中一般都不止一个业务系统，不同的业务系统或多或少都会有消息推送的需求，不论是对内还是对外，通常 情况下消息的推送都与业务系统耦合在一起实现，没法做到复用，如果另外一个业务系统也需要有消息推送功能，又得重新对接一遍， 无形之中增加工作量，我所在的企业就是有这种需求，目前实现也是耦合在一起，而且跟业务强关联，不同得业务系统的消息推送规则 不一样，但是底层的后台推送功能确实大同小异，故此，为了节约成本，降低耦合，有了这篇通用的消息推送设计解决方案小册。

# 二、 本质

消息推送的本质: “某种条件”满足触发消息推送，从消息源端到消息推送端再到接收端，消息的流转总共要经过三个平台才能 触达到接收对象，接收对象可以是用户手机、用户邮箱、App等

# 三、推送方式

目前比较主流的推送方式有短信推送、电话通知、App推送、企业微信和邮件推送等，具体定义如下：

  • 短信推送
    借助手机的短信功能发送信息，这种即使用户已经卸载了软件还是可以发送到用户手机上，但成本相对较高，业务系统通过消息中心将消息 发送到用户手机短信上
  • 电话通知
    电话语音通知，业务系统通过消息中心将消息以电话语言的方式通知用户
  • App推送
    App推送也叫站内推送，实际是通过App对软件中的注册用户发送信息，用户只有打开App才能看到，好处是不需要外部成本，坏处是传播能力不足
  • 微信推送
    通常是指企业微信，业务系统通过消息中心将消息推送给企业微信
  • 邮件推送
    电子邮件发送,业务系统通过消息中心将消息推送给用户电子邮箱

本小册，只讲解短信推送及实现，其它几种留给各位同学实现。

# 四、推送规则

业务业务推送规则，包括推送方式、推送时间、推送间隔、是否启用等，总之，需满足消息准确推送到业务系统

# 五、推送客户端

  • 生产端
    生产消息，将消息按一定的规则保存起来
  • 接收端
    监听到消息，然后处理消费，判断消息推送平台
  • 发送端
    第三方消息推送平台（例如：阿里云、极光等）
